# Recommended layout — tj-shop v1 backend

Synthesized from `reference-repos-2026-05-15.md`. The shape is: koskeller's skeleton +
launchbadge/sqlx's `#[sqlx::test]` tests + a couple of dep-version updates
from sheroz. Single crate, no workspace, no auth yet.

Deployment assumption (folded in): **monorepo with TS React frontend, both
served from the same small VPS, fronted by Caddy.** See "Deployment topology"
below. The backend stays a pure JSON server; Caddy serves the React build
and proxies `/api/*` to Rust.

## Repo layout (monorepo)

```
tj-shop/
├── backend/                    # Rust crate — this doc's subject
│   ├── Cargo.toml
│   ├── .env.sample
│   ├── .sqlx/
│   ├── migrations/
│   ├── src/
│   └── tests/
├── frontend/                   # Vite + TS + React (separate work)
│   ├── package.json
│   ├── vite.config.ts
│   └── src/
├── Caddyfile                   # prod: Caddy fronts both apps on the VPS
├── compose.yaml                # dev only: containerized Postgres on laptop
├── tj-shop.service             # prod: systemd unit for the Rust binary
├── README.md
├── docs/
│   └── claude/                 # this research
└── v0/                         # archived static app
```

- **Two functional dirs.** `backend/` and `frontend/` are the only places
  you'll spend development time. Everything else is small and lives at the
  top level as bare files (Caddyfile, systemd unit, README) or pre-existing
  directories (`docs/`, `v0/`).
- **No Cargo workspace.** `backend/` is a standalone crate; there's no
  second Rust crate yet, so a workspace would be ceremony. Reconsider if you
  add a CLI or worker.
- **`backend/` and `frontend/` are siblings**, not nested. Each can be opened
  in its own editor window with its own toolchain. CI can change-detect per
  subdir cheaply.
- **`Caddyfile`, `compose.yaml`, and `tj-shop.service` live at the root**,
  not under a `deploy/` subdir. They're one file each. `compose.yaml` spins
  up the dev Postgres on your laptop; `Caddyfile` and `tj-shop.service`
  configure the prod VPS. Splitting them into subdirs would be premature
  structure for three files.
- All paths below in this doc are relative to **`backend/`** unless noted.

### Module rationale

- **`main.rs`** — boot only. Load `.env`, init tracing, build `Config`, build
  `PgPool`, run migrations, build `Router`, call `axum::serve(...)
  .with_graceful_shutdown(...)`. Anything more goes elsewhere.
- **`lib.rs`** — defines `AppState` and `pub fn router(state: AppState) ->
  Router`. The test harness and `main.rs` both call `router()`, which means
  there is exactly one place where the router is wired.
- **`config.rs`** — env vars in, `Config` struct out. Returns `Result<Config,
  ConfigError>` so `main.rs` can fail fast with a helpful message.
- **`db.rs`** — thin wrapper around `PgPoolOptions`: `pub async fn pool(cfg:
  &Config) -> sqlx::Result<PgPool>` and `pub async fn migrate(pool: &PgPool) ->
  sqlx::Result<()>`. Do not invent a `Db` struct — `PgPool` is already an
  Arc'd handle, naming it once is enough.
- **`error.rs`** — `ApiError` enum, `IntoResponse` impl, `pub type Result<T> =
  std::result::Result<T, ApiError>;`. Handlers return `Result<Json<T>>`.
- **`telemetry.rs`** — `pub fn init_tracing()` (registry + EnvFilter + fmt
  layer; JSON in prod, pretty in dev) and `pub fn trace_layer() ->
  TraceLayer<...>` (request-level spans). Two factories, called from
  `main.rs` and `router()` respectively.
- **`middleware.rs`** — factory functions only. `request_id_layer()`,
  `timeout_layer()`, `normalize_path_layer()`, and `cors_layer()` (the last
  applied **only when `APP_ENV=dev`** — see "Deployment topology"). Each
  returns the concrete tower layer; `router()` stacks them. Keeps `lib.rs`
  short.
- **`routes/`** — one file per resource. Each file exports `pub fn router()
  -> Router<AppState>`. `routes/mod.rs` merges them. Handlers in the same file
  as the route; no separate `handlers/` directory. Inline `sqlx::query!`
  calls in handlers are fine at this scale — extract to `db.rs` only if a
  query is used by more than one handler.
- **`tests/`** — one integration-test file per resource, each using
  `#[sqlx::test]` to get a fresh per-test database. `tests/common/mod.rs`
  holds the request/response helpers.

## Error type

**Use `thiserror` for the enum, `anyhow` only as a `#[from]` variant for
fall-through context.** Concretely:

```rust
// src/error.rs
use axum::{response::{IntoResponse, Response}, Json, http::StatusCode};
use thiserror::Error;

#[derive(Error, Debug)]
pub enum ApiError {
    #[error("not found")]
    NotFound,

    #[error("invalid request: {0}")]
    BadRequest(String),

    #[error("conflict: {0}")]
    Conflict(String),

    #[error("database error")]
    Sqlx(#[from] sqlx::Error),

    #[error("internal error")]
    Anyhow(#[from] anyhow::Error),
}

impl IntoResponse for ApiError {
    fn into_response(self) -> Response {
        // Always log the real error before sanitizing.
        tracing::error!(error = ?self, "request failed");
        let status = match &self {
            ApiError::NotFound        => StatusCode::NOT_FOUND,
            ApiError::BadRequest(_)   => StatusCode::BAD_REQUEST,
            ApiError::Conflict(_)     => StatusCode::CONFLICT,
            ApiError::Sqlx(_)         => StatusCode::INTERNAL_SERVER_ERROR,
            ApiError::Anyhow(_)       => StatusCode::INTERNAL_SERVER_ERROR,
        };
        (status, Json(serde_json::json!({"error": self.to_string()}))).into_response()
    }
}

pub type Result<T, E = ApiError> = std::result::Result<T, E>;
```

**Why this and not alternatives:**

- *Not* `anyhow::Error` directly in handlers. It would compile (with
  `axum-extra`/glue), but loses the explicit status mapping that distinguishes
  a 404 from a 500. Every reference repo surveyed uses a `thiserror` enum.
- *Not* a generic `AppError` that wraps `Box<dyn Error>`. The whole point of
  the enum is exhaustive `match`.
- The flat `{ "error": "..." }` JSON body matches koskeller and launchbadge/sqlx.
  Sheroz's RFC 7807-ish envelope is overkill until a frontend asks for it.

## AppState

```rust
// src/lib.rs
#[derive(Clone)]
pub struct AppState {
    pub db: sqlx::PgPool,
    pub config: std::sync::Arc<crate::config::Config>,
}
```

- `PgPool` is internally `Arc`'d; cloning it is cheap. No extra wrapper.
- `Config` is wrapped in `Arc` because (a) it's slightly larger and (b)
  cloning a `String`-heavy struct on every request is wasteful. Following
  koskeller.
- The struct is `Clone`, not `Arc<AppState>`. Axum's `State` extractor
  clones the state per request, which is fine when all fields are cheap to
  clone.

Pass into the router via `.with_state(state)` in `lib.rs::router()`.

## Migration strategy

**Run on startup, in `main.rs`, before `axum::serve`.** Use
`sqlx::migrate!("./migrations").run(&pool).await?`.

- Same binary, no separate migration command. For a single-instance app,
  this is the right default and matches every reference repo.
- The `_sqlx_migrations` table tracks state; reruns are no-ops.
- Migrations are checked into git, named `YYYYMMDDHHMMSS_<slug>.sql`
  (what `sqlx migrate add` produces).
- **Adopt launchbadge's schema conventions:** every table has `created_at
  timestamptz not null default now()` and `updated_at timestamptz not null
  default now()`, with the `set_updated_at()` / `trigger_updated_at()`
  helpers from `launchbadge/realworld-axum-sqlx/migrations/1_setup.sql` in
  the first migration. Native Postgres `uuid` type with `uuid_generate_v1mc()`,
  not `CHAR(36)`.

**When you outgrow this:** if you ever deploy >1 instance, move the
`migrate!()` call into a separate `bin/migrate.rs` and run it as a
pre-deploy step. Not a v1 concern.

## Config loading

**Hand-rolled with `std::env::var` + `dotenvy` for development.** Same as
koskeller. No `config` crate, no `figment`, no `envy`.

Reasons:
- Total config surface for v1 is ~5 vars (`DATABASE_URL`, `PORT`, `RUST_LOG`,
  `APP_ENV`, maybe a CORS origin). A 30-line hand-rolled reader is shorter
  than the import statements for `config`.
- Hand-rolled gives explicit, actionable panic messages on missing vars.
- `dotenvy::dotenv().ok()` at the top of `main.rs` for local dev; in
  production, env vars come from the deployment platform.

```rust
// src/config.rs (shape, not literal)
pub struct Config {
    pub database_url: String,
    pub listen_addr: SocketAddr,
    pub db_pool_max: u32,
    pub app_env: AppEnv,
}
pub enum AppEnv { Dev, Prod }

impl Config {
    pub fn from_env() -> Result<Self, ConfigError> { /* parse + validate */ }
}
```

## Logging / tracing

**`tracing` + `tracing-subscriber` + `tower-http::trace::TraceLayer`.** The
stack everyone uses.

- `tracing-subscriber::registry().with(EnvFilter::try_from_default_env()
  .unwrap_or_else(|_| "info,tower_http=info,sqlx=warn".into())).with(fmt_layer).init()`
- Format layer: `fmt::layer().json()` in production, `fmt::layer().pretty()`
  in dev, switched on `Config::app_env`.
- HTTP span layer: `TraceLayer::new_for_http().make_span_with(DefaultMakeSpan::new()
  .level(Level::INFO))` applied as the *outermost* layer in `router()`.
- Request IDs via `tower_http::request_id::SetRequestIdLayer` with a
  `MakeRequestId` impl that emits `Uuid::now_v7()` — copied from koskeller.
  Pair with `PropagateRequestIdLayer` so the response echoes the header.

## Test strategy

**`#[sqlx::test]` against a real Postgres, one fresh database per test.**

- Requires `DATABASE_URL` to point at a Postgres where the test runner has
  `CREATEDB` privileges. Local dev: the Postgres started by `compose.yaml`
  (see "Local Postgres via Docker"); the `POSTGRES_USER` from the official
  image is a superuser, so CREATEDB works out of the box.
- `#[sqlx::test]` automatically runs migrations from `./migrations/` into
  the fresh database before injecting the `PgPool`.
- The handler under test is exercised via `tower::ServiceExt::oneshot` on
  the `Router` returned by `lib.rs::router()` — no real TCP listener.
- For seed data, use `#[sqlx::test(fixtures("products"))]` and put SQL
  fixtures under `tests/fixtures/`.

Skip:
- testcontainers (overkill for v1; `#[sqlx::test]` already gives isolation)
- a shared test pool with rollback-per-test (faster but loses migration
  testing)
- mock the database (you'd lose 90% of the value of integration tests; if
  you want unit tests, write them against pure functions in `db.rs`)

Why this and not the koskeller `create_test_db` approach: `#[sqlx::test]`
is fewer lines, official, and does cleanup. Koskeller predates wide
`#[sqlx::test]` adoption.

Why each:

  `tower-http` features pinned to the minimum needed; add more as you go.

Deps deliberately omitted from v1:
- **utoipa / aide** — see open-questions.
- **validator** — defer until you have non-trivial validation needs;
  `BadRequest(String)` from a manual check is fine for now.
- **redis** — none of the features you've described need it.
- **jsonwebtoken / argon2** — auth is out of scope.
- **figment / config** — see config rationale above.

## Deployment topology

Single small VPS, everything co-located. Boring stack:

```
Internet ──TLS──▶ Caddy (:443)
                   ├── /api/*  ──▶  Rust  (127.0.0.1:3000)  ──▶  Postgres (127.0.0.1:5432)
                   └── /*      ──▶  static React build      (/var/www/tj-shop/)
```

- **Caddy** terminates TLS (auto-cert via Let's Encrypt), serves the React
  build, and reverse-proxies `/api/*` to the Rust process on loopback.
  Boring, single config file, no Nginx + certbot juggling.
- **Rust binds `127.0.0.1:3000`**, not `0.0.0.0`. External traffic only
  reaches it through Caddy. One fewer way to leak.
- **Postgres on the same box**, listening on loopback, **shared across all
  apps on the VPS**. `apt install postgresql-16` once; for each app, create
  one database and one role that owns it. tj-shop gets `tj_shop` database +
  `tj_shop` role; the next app gets its own pair. `DATABASE_URL` for
  tj-shop is `postgres://tj_shop:...@localhost/tj_shop`. No PgBouncer at
  this scale; `PgPoolOptions::max_connections(10)` per app — Postgres
  default `max_connections=100`, so you fit ~10 apps before tuning.
- **systemd** supervises the Rust binary. `EnvironmentFile=/etc/tj-shop.env`
  carries `DATABASE_URL` etc.; `Restart=on-failure`; stdout/stderr go to
  journald.

Static-serving fork: an alternative is to let Rust serve the React build
itself (`tower-http::services::ServeDir` + SPA fallback, plus
`tower-http/fs` in deps). One fewer process, one binary owns routing. I'm
recommending Caddy-serves-static because (a) the user framed the backend
as "pure JSON," (b) frontend and backend can deploy independently, and (c)
Caddy already exists in the topology for TLS — making it also serve files
is free. Reverse if you want truly single-binary deploys.

### Root-level infra files

**`Caddyfile`** (shape, not literal):

```
tj-shop.example.com {
    encode zstd gzip
    handle /api/* {
        reverse_proxy 127.0.0.1:3000
    }
    handle {
        root * /var/www/tj-shop
        try_files {path} /index.html
        file_server
    }
}
```

**`tj-shop.service`** (shape — symlink or copy into `/etc/systemd/system/`
on the VPS):

```
[Unit]
Description=tj-shop backend
After=network.target postgresql.service
Requires=postgresql.service

[Service]
Type=simple
User=tj-shop
EnvironmentFile=/etc/tj-shop.env
ExecStart=/usr/local/bin/tj-shop
Restart=on-failure
RestartSec=2

[Install]
WantedBy=multi-user.target
```

### Dev workflow (consequence of the above)

Three processes in dev — Postgres in a container, backend and frontend
native:

0. **Postgres (Docker)**: `docker compose up -d` from the repo root.
   Detached, one-time per session. See "Local Postgres via Docker" below.
1. **`backend/`**: `cargo run` — Rust on `127.0.0.1:3000`, talks to the
   container's Postgres on `127.0.0.1:5432`.
2. **`frontend/`**: `npm run dev` — Vite on `127.0.0.1:5173`, configured
   with a proxy so `/api/*` requests forward to `:3000`. **Vite and Rust
   share an origin from the browser's point of view in dev**, just like
   prod through Caddy.

This proxy approach is the reason the `cors_layer()` is only wired when
`APP_ENV=dev` and even then only as a safety net — with the Vite proxy,
the browser never sees `:3000` directly, so CORS shouldn't fire in normal
dev either. The dev-mode CORS layer exists for the case where someone hits
`:3000` from a tool that doesn't proxy.

In production, the React build is `npm run build`-ed to `frontend/dist/`,
then deployed (rsync / git pull + build on box) to `/var/www/tj-shop/`.
Caddy serves it; same origin as the API; no CORS layer in the request
path at all.

## Local iteration ↔ VPS parity

Stated assumption: you iterate on this app **locally** (laptop) and deploy
the same code to a **minimal VPS**. The layout is designed around that pair
— no staging, no CI required for v1. **Postgres runs in a Docker container
locally and natively on the VPS** (see "Local Postgres via Docker" below
for the rationale).

What has to match between the two boxes:

- **Postgres major version.** The `postgres:16` image tag locally must
  match the `postgresql-16` package version on the VPS. Same SQL, same
  type behaviour, same migration semantics. Bump both together when you
  upgrade.
- **Env-var contract.** Same set of variables in both environments; only
  values differ. Locally: `backend/.env` loaded by `dotenvy`. On the VPS:
  `EnvironmentFile=/etc/tj-shop.env` loaded by systemd. The same
  `Config::from_env()` reads both.
- **Migrations are auto-applied on startup, everywhere.** `sqlx::migrate!()`
  is idempotent: first startup on the VPS applies all pending migrations;
  subsequent startups no-op. Same on your laptop after `docker compose up`.

What's intentionally different:

- **Postgres runtime: container locally, native on the VPS.** Same major
  version, same SQL — different supervisor. See "Local Postgres via Docker"
  for why this asymmetry is the right call.
- **No Caddy, no systemd locally.** Locally: `cargo run` in `backend/` and
  `npm run dev` in `frontend/`. On the VPS: Caddy fronts, systemd supervises.
  This divergence is intentional — those are deployment concerns, not dev
  concerns, and they add nothing to the inner-loop experience.
- **CORS in dev only.** Already covered in "Deployment topology" → "Dev
  workflow." Worth restating because it's the one runtime code path that
  branches on `APP_ENV`.

### Local Postgres via Docker

**Only the database is containerized locally**; the Rust backend and
frontend run natively. This buys you one-command, version-pinned Postgres
without paying the Docker daemon RAM tax on the VPS.

**`compose.yaml`** (shape, not literal — at repo root):

```yaml
services:
  db:
    image: postgres:16
    ports:
      - "5432:5432"
    environment:
      POSTGRES_USER: tj_shop
      POSTGRES_PASSWORD: tj_shop
      POSTGRES_DB: tj_shop
    volumes:
      - db_data:/var/lib/postgresql/data
    restart: unless-stopped

volumes:
  db_data:
```

- `backend/.env.sample` ships
  `DATABASE_URL=postgres://tj_shop:tj_shop@localhost:5432/tj_shop`.
- Dev credentials are hardcoded in `compose.yaml` — they're not secrets,
  the container only listens on loopback via the host port mapping.
- Data persists in a named Docker volume (`db_data`), managed by Docker,
  not the repo. Wipe with `docker compose down -v` for a clean slate.
- **Port-collision note:** if you already have Postgres on `:5432`
  (Homebrew, Postgres.app), either stop it or remap to `5433:5432` in
  compose and update `DATABASE_URL` to match.

**Why container in dev, native in prod:**
- *Dev*: laptops have plenty of RAM; the ~100–150 MB Docker daemon
  overhead is invisible. The win is one-command Postgres with version
  pinning and trivial reset.
- *Prod*: minimal VPS doesn't have RAM to spare, and Postgres-in-Docker
  buys nothing — `apt install postgresql-16` is just as version-pinned
  and ~200 MB cheaper.

### Multiple apps sharing the VPS Postgres

This is the standard pattern; Postgres is multi-tenant by design. When you
add a second app:

- **One Postgres install on the VPS, one database per app, one role per
  app.** tj-shop owns `tj_shop` database via the `tj_shop` role; app#2
  owns `app2` database via the `app2` role. Roles have no privileges on
  databases they don't own.
- **Each app's `sqlx::migrate!()` scope is its own database.** No
  cross-app interference.
- **Each app gets its own systemd unit** (`tj-shop.service`, `app2.service`).
  Both `Requires=postgresql.service` so they don't start before Postgres
  is up.
- **Connection budget.** Default `max_connections=100`; with
  `PgPoolOptions::max_connections(10)` per app, you fit ~10 apps before
  needing to raise the ceiling.
- **Caddy gains a second site block** (or a second `handle_path` inside
  one) for the new app's domain or path prefix. Static and proxy work the
  same way.
- **Backups become one `pg_dumpall`** for everything, or per-database
  `pg_dump <dbname>` for granular restore. Out of scope for this doc;
  flag for the deploy runbook.

The asymmetry that matters: **only Postgres is shared**. Each app keeps
its own binary, its own systemd unit, its own Rust crate, its own repo
or monorepo subdir. The shared resource is the database server, nothing
else.

For local dev across multiple apps: easiest is to keep each app's
`compose.yaml` running on its own host port (`5433`, `5434`, ...) and
update each app's `DATABASE_URL` accordingly. Consolidate into one shared
laptop-wide Postgres only if managing N containers becomes annoying.

### Deploy mechanic

Three boring options. Pick by VPS spec, not by preference:

1. **`git pull && cargo build --release && systemctl restart tj-shop`**
   on the box. No toolchain skew, fully self-contained, one command.
   *Requires* ~4 GB RAM and several minutes of compile per deploy. Fine
   on a mid-range VPS; will OOM-kill on a 1 GB box.
2. **Cross-compile locally, rsync the binary.** On macOS, `cargo-zigbuild`
   targeting `x86_64-unknown-linux-gnu` (or `-musl`) is the least-friction
   path. Fast deploys, but you set up a cross toolchain once.
3. **GitHub Actions builds, you `scp` (or `gh release download`) the
   artifact.** Overkill for v1 but trivial to add later — same binary, just
   built somewhere else.

**Default for v1:** option 1 if the VPS has ≥4 GB RAM; option 2 if you're
deploying to a 1–2 GB instance. Either way, **the deployable is a single
binary plus the `migrations/` directory** — copy those two paths, restart
the service, done. Frontend is independent: `npm run build` in `frontend/`,
rsync `frontend/dist/` to `/var/www/tj-shop/`.

### Implication for the TDD cadence

If you commit to test-driven development (see prior conversation), the
local box is where the red-green loop happens — `cargo test` against local
Postgres, fast feedback. The VPS doesn't run tests; it only serves
already-tested binaries. This means **your laptop's Postgres setup is
load-bearing for development velocity** — don't skip pinning its version.

## Decisions baked into this layout

These four are the load-bearing choices. Worth restating:

1. **Single crate, single binary, migrations run in `main.rs`.** Reverse
   this only if you actually need a second deployable.
2. **`thiserror` enum + `IntoResponse`, with both `Sqlx(#[from])` and
   `Anyhow(#[from])` variants.** Every surveyed repo converges on this.
3. **`#[sqlx::test]` for integration tests.** Real database, per-test
   isolation, official. Skip mocking and skip test containers.
4. **Caddy fronts everything in prod; Rust is pure JSON on loopback;
   Postgres is shared infra — native on the VPS, containerized locally.**
   Prod deployment is one VPS, one Caddy config, one Postgres install
   (shared across apps via separate databases + roles), one systemd unit
   per app. Dev mirrors it except: native frontend/backend processes (no
   Caddy/systemd) and Postgres via `compose.yaml`.
